# 차지비 상면관리자 페이지 (chargev-admin) 중간 보고

> 부문장 보고용 | 작성일 2026-06-12 | 작성자 lodi

---

## TL;DR (한 줄 요약)

본사 운영팀과 17,000 충전소 파트너가 함께 쓸 **통합 운영 콘솔**을 자체 구축 중.
모든 외부 의존성을 **어댑터 패턴**으로 추상화해 사내 시스템(admin, BPMS, Zendesk 등)과 갈아끼울 수 있도록 설계.
핵심 운영 흐름은 99% 완성, **남은 작업은 사내 시스템과의 실제 API 연동**.

---

## 1. 프로젝트 배경

### 무엇을 만드나
- 본사 운영팀과 17,000 충전소 파트너가 함께 사용하는 통합 관리자 콘솔
- 사내 admin은 그대로 유지, 그 위에 파트너 노출 + 운영팀 보조 도구를 추가

### 왜 만드나
| 문제 | 해결 방향 |
|---|---|
| 운영 도구가 사내 여러 곳에 분산됨 (계정, 충전소, 정산, CS, 점검, 공지) | 운영팀에게는 통합 허브 |
| 17,000 파트너가 본인 충전소 정보를 보는 일관된 진입점이 없음 | 파트너 전용 공개 콘솔 |
| 사내 admin은 내부 직원 전용으로 설계됨 (파트너 노출 불가) | 파트너 영역만 별도 콘솔로 분리 |

### 우리 콘솔과 사내 admin의 관계

```
┌─────────────────────────────────────────────────────────┐
│  사내 admin (이미 운영 중, 유지)                          │
│  내부 직원 일상 운영                                       │
│  충전기, 충전소, B2C 회원, 충전 이력, OCPP 상태             │
└──────────────────────┬──────────────────────────────────┘
                       │ read-only API
                       ↓
┌─────────────────────────────────────────────────────────┐
│  chargev-admin (신규)                                    │
│  파트너 17,000명에게 노출되는 콘솔                          │
│  운영팀 보조 도구 (파트너 계정/권한/감사로그/메시징)         │
└──────────────────────┬──────────────────────────────────┘
                       │
       ┌───────────────┼────────────────┐
       ↓               ↓                ↓
   BPMS(SF)        Zendesk          NICE/비즈메시지
   계약 정보       CS 문의/티켓      본인인증/발송
```

## 2. 핵심 설계 컨셉

### 5가지 원칙

| # | 원칙 | 의도 |
|---|---|---|
| 1 | **어댑터 패턴** | 본인인증, 사내 admin, BPMS, Zendesk, 파일 저장, 알림 모두 mock으로 추상화. 사내 시스템 확정되면 구현체 1개만 갈아끼움 |
| 2 | **Source of Truth 매트릭스** | "어떤 데이터를 누가 책임지는지" 명확히 분리. 자체 CMS 만들지 않음 (파편화 방지) |
| 3 | **3-Layer 디자인 토큰** | BI 한 곳만 수정하면 전체 UI 따라옴. 추후 BI 변경 대비 |
| 4 | **B2B 단순화** | 셀프 가입 폐기. 운영자가 직접 계정 발급 후 메일/SMS/카카오로 안내. 17,000명 검증 부담 0 |
| 5 | **Role 템플릿 + 추가 권한 Grant** | 3단계 role(최고/중간/말단)이 기본 템플릿. 매출 같은 민감 정보는 사용자별로 추가 grant |

## 3. 진행 현황

### ✅ 완료 (운영 가능 수준)

**인증과 보안**
- 세션 관리, 비밀번호 정책, 계정 잠금, 2FA UI, 강제 비번 변경
- 감사 로그 (모든 write 작업 자동 기록)
- 권한 시스템 (role 템플릿 + 12개 permission 키 + 사용자별 grant UI)

**운영자 백오피스**
- 신규 계정 등록 + 임시비번 자동 발송 (메일/SMS/카카오)
- 계정 잠금, 권한 변경, 비밀번호 재발급, 강제 로그아웃
- 충전소별 권한 매핑 (Owner/Member/Viewer)
- 메일/문자 일괄 발송 도구
- 충전소 그룹 관리 (분류/태깅)

**파트너 핵심 화면**
- 충전소 리스트, 단일 충전소 상세, 충전기 통합 테이블, 실시간 모니터링
- 본인 충전소만 자동 필터링

**운영 일상 도구**
- 유지보수 티켓 접수/처리
- 점검 일정 관리
- 공지사항, 자료게시판
- 정산 (이력, 한전 납부, 그룹 관리)

### ⏸ 보류 (별도 결정 필요)

| 항목 | 사유 |
|---|---|
| AI 챗봇, AI 리포트 | 방향성 별도 검토 필요. 메뉴 임시 비활성화 |
| 계약 정보 화면 | Salesforce 연동 정해진 다음 진행 |
| 알림 센터 (인앱) | 채널과 발송 정책 정해진 다음 진행 |

## 4. 사내 시스템 매핑 (Source of Truth)

### 데이터별 SoT

| 데이터 | SoT | 우리 콘솔 역할 |
|---|---|---|
| 충전기, 충전소 메타 | 사내 admin | 어댑터로 조회 표시 |
| B2C 회원 (충전 사용자) | 사내 admin | 직접 표시 안 함 (충전 이력 표시 시 anonymize) |
| 충전 이력 로그 | 사내 admin | 정산 모듈에서 호출 |
| 충전기 실시간 상태 (OCPP) | 사내 admin | 실시간 모니터링에서 폴링 |
| 계약 정보 | BPMS (Salesforce) | 충전소 상세에서 표시 + 만료 알림 |
| CS 문의, 유지보수 티켓 | Zendesk | 표시 + 접수 (생성/상태 동기화) |
| **파트너 콘솔 계정** | **chargev-admin** | 직접 책임 (CRUD) |
| **콘솔 권한 (role, grant)** | **chargev-admin** | 직접 책임 |
| **충전소 ↔ 파트너 매핑** | **chargev-admin** | 직접 책임 |
| **콘솔 감사 로그** | **chargev-admin** | 직접 책임 |
| **콘솔 공지, 자료게시판** | **chargev-admin** | 직접 책임 |
| **콘솔 점검 일정** | **chargev-admin** | 직접 책임 (Zendesk와 별개) |
| 본인인증 | NICE (사내 표준) | 어댑터로 호출 |
| SMS/카카오 알림톡 발송 | 사내 비즈메시지 | 어댑터로 호출 |
| 파일 (증빙, 자료) | 사내 NAS or S3 | 어댑터로 업로드/다운로드 |

### 핵심 결정 (확정된 4가지)

1. **회원정보(B2C 충전 사용자)** = 사내 admin SoT. 우리 콘솔 파트너 계정과 완전 분리
2. **파트너(콘솔 계정)** = 우리가 직접 SoT. 운영자가 생성/발급
3. **유지보수 티켓** = Zendesk가 SoT. 우리 콘솔에선 표시/접수 인터페이스만 (자체 모델은 폐기 또는 캐시로 축소)
4. **정산 데이터, 실시간 OCPP** = 사내 admin SoT. 어댑터로 가져옴

## 5. 필요한 외부 API (담당팀에 요청 사항)

### 5-1. 사내 admin API (가장 큰 의존성)

| 용도 | API 형태 |
|---|---|
| 충전소 목록 조회 | `GET /stations` (페이지네이션, 필터) |
| 충전소 단건 조회 | `GET /stations/{id}` |
| 충전소 검색 | `GET /stations?search=` (이름/주소/ID) |
| 충전기 목록 (by 충전소) | `GET /stations/{id}/chargers` |
| 충전기 실시간 상태 | `GET /chargers/realtime?ids=` 또는 SSE/WebSocket |
| 충전 세션 (진행 중) | `GET /charge-sessions/active` |
| 충전 이력 | `GET /charging-history?from=&to=&stationIds=` |
| 가동률/매출 집계 | `GET /stations/{id}/metrics?period=` |

확인 필요: **인증 방식** (API key, OAuth, 사내망 IP whitelist 등)

### 5-2. BPMS (Salesforce) API

| 용도 | API 형태 |
|---|---|
| 계약 조회 (by 사업자번호) | `GET /contracts?businessNo=` |
| 계약 조회 (by 충전소) | `GET /contracts?stationId=` |
| 만료 임박 계약 | `GET /contracts?expiringWithinDays=90` |
| 계약 문서 다운로드 | `GET /contracts/{id}/document` |

확인 필요:
- Salesforce REST API 접근 권한 (Connected App, OAuth)
- 사용 sObject 매핑 (Account, Contract, custom object 등)

### 5-3. Zendesk API

| 용도 | API 형태 |
|---|---|
| 티켓 목록 | `GET /api/v2/tickets` |
| 파트너별 티켓 | `GET /api/v2/users/{id}/tickets/requested` |
| 충전소별 티켓 | custom field 또는 tag로 필터 (`tags=station:CS001`) |
| 신규 티켓 접수 | `POST /api/v2/tickets` |
| 상태 변경/답변 | `PUT /api/v2/tickets/{id}` |
| 티켓 코멘트 | `GET /api/v2/tickets/{id}/comments` |

확인 필요:
- Zendesk API 토큰 (관리자 또는 전용 봇 계정)
- 충전소 ↔ 티켓 연결 방식 합의 (custom field, tag, 또는 별도 매핑 테이블)
- Zendesk 요청자 ↔ 우리 콘솔 파트너 매핑 정책

### 5-4. NICE 본인인증

확인 필요: 사내에서 NICE를 어떤 방식으로 사용 중인지 (라이브러리, SaaS, 자체 wrapper)

### 5-5. SMS / 카카오 알림톡

확인 필요: 사내 비즈메시지 채널 정보 (벤더, 사업자 정보, API 형태)

### 5-6. 파일 스토리지

확인 필요: 사내 NAS 경로 또는 S3 버킷 + 접근 방식 (마운트, 인증 등)

## 6. 코드 조정 영향 (이미 만든 것 중)

| 자체로 만든 것 | 조정 방향 |

| 정산 모듈 (기존) | **유지** — 데이터만 사내 admin에서 가져오게 어댑터 정리 |

## 7. 추가 결정 필요 사항

1. **Zendesk 티켓 ↔ 충전소 연결 방식**
   - 옵션 A: Zendesk custom field에 `station_id` 추가 (Zendesk 설정 필요)
   - 옵션 B: Zendesk 태그로 `station:CS001` 부여
   - 옵션 C: 우리 DB에 별도 매핑 테이블 (`ZendeskTicketLink`)
2. **B2C 회원과 파트너 분리** 확정 시 — 우리 콘솔에서 충전 이력 표시 정책 (anonymize 수준, 마스킹)
3. **NICE 본인인증 사용 시점** — 운영자가 파트너 등록 시? 파트너 비번 변경 시?
4. **AI(챗봇, 리포트)** 방향성 — 자체 모델 / OpenAI API / 사내 AI 표준
5. **운영 DB** — 현재 PostgreSQL 16. 사내 AWS DB (RDS/Aurora 등) 확정 필요

## 10. 데모 시나리오 (보고 시 시연 제안)

| 데모 | 의도 |
|---|---|
| 최고 관리자(qq@qq.q)로 로그인 → 모든 메뉴 노출 | 운영팀 전체 뷰 |
| 중간 관리자(aa@aa.a)로 로그인 → 매출/감사로그 메뉴 보임 | role + 추가 grant 동작 |
| 말단 직원(zz@zz.z)으로 로그인 → 매출/정산/계정관리 hidden | 권한 분리 |
| 운영자 화면에서 신규 파트너 계정 발급 → 임시비번 자동 발송 (콘솔 로그) | 17,000 onboarding 흐름 |
| 파트너(partner@example.com)로 로그인 → 본인 충전소만 보임 + 티켓 접수 | 파트너 일상 사용 |
| 충전소 상세 → 실시간 모니터링 (10초 폴링, 충전 진행률) | 모니터링 도구 |

시드 계정 (3개 모두 비번 `qwe123!@#`):
- `qq@qq.q` 최고 관리자
- `aa@aa.a` 중간 관리자
- `zz@zz.z` 말단 직원
